SQL · 01 SQL

Sintaxe Básica Executiva

Databricks / Spark SQL

O que é

A sintaxe ANSI fundamental. Para um engenheiro júnior recém-chegado, o SQL pode ser intuitivo de ler, mas a ordem de execução nos bastidores obedece a regras muito rígidas. Filtrar o resultado de uma equação na cláusula errada retorna erros ou faz o Spark varrer a base à toa.

Júnior

Ordem de Execução e o Esqueleto Clássico

A forma como escrevemos é diferente da forma como o engine lê.

Ordem que escrevemos: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY
Ordem que o engine processa: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

SQL 
-- O Esqueleto Completo
SELECT
    categoria_id,              -- 5º processado
    SUM(valor) AS receita
FROM tabela_master             -- 1º: a origem principal
WHERE status_venda = 'FECHADA' -- 2º: corta lixo ANTES de calcular
GROUP BY categoria_id          -- 3º: empacota linhas filtradas
HAVING SUM(valor) > 1000       -- 4º: filtra RESULTADOS das somas
ORDER BY receita DESC          -- 6º: ordena visualmente
LIMIT 10;                      -- 7º: corta para os top 10

Operadores Lógicos Críticos

SQL 
SELECT *
FROM banco_central
WHERE
    sigla_estado IN ('SP', 'RJ', 'MG')               -- nunca use OR encadeado
    AND data_nascimento BETWEEN '2000-01-01' AND '2000-01-05'  -- inclui bordas

DISTINCT e Boas Práticas Operacionais

⚠ SELECT DISTINCT * é uma das operações de Hash Shuffle mais pesadas num cluster massivo. Deduplique cirurgicamente, nunca em toda a tabela.
SQL 
-- ❌ EVITE
SELECT DISTINCT * FROM fato_gigantesca;

-- ✅ BOM: DISTINCT cirúrgico em chaves atômicas
SELECT DISTINCT id_cliente
FROM clientes_ativos
WHERE id_cliente IS NOT NULL;

Operadores de Conjunto (Set Operations)

SQL 
-- UNION ALL: empilha mantendo duplicatas (mais rápido)
SELECT id_user FROM base_antiga
UNION ALL
SELECT id_user FROM base_nova;

-- EXCEPT: o que existe em A e NÃO existe em B
SELECT id_transacao FROM extrato_banco
EXCEPT
SELECT id_transacao FROM base_conciliada;

-- INTERSECT: somente o que existe em ambas
SELECT id_produto FROM catalogo_site
INTERSECT
SELECT id_produto FROM estoque_fisico;

Protegendo a Tipagem com TRY_CAST

Quando a origem vem suja, CAST(...) pode abortar toda a execução. Prefira TRY_CAST(...) para transformar valores inválidos em NULL e tratar isso explicitamente depois.

SQL 
SELECT
    TRY_CAST(idade_txt AS INT)     AS idade_segura,
    TRY_CAST(flag_ativo_txt AS BOOLEAN) AS is_ativo
FROM camada_bronze;

-- Valores monetários: nunca use FLOAT ou DOUBLE
SELECT
    CAST(vlr_saldo AS DECIMAL(16,4))   AS saldo_financeiro,
    TRY_CAST(imposto AS DECIMAL(6,2)) AS aliquota
FROM extrato_banco;
Sênior

Parsing de Datas e Timezone com Regra Explícita

⚠ O erro mais comum não é a função em si — é misturar timestamps armazenados em UTC com sessões configuradas em outro fuso sem conversão explícita.
SQL 
SET TIME ZONE 'America/Sao_Paulo';

SELECT
    from_utc_timestamp(ts_evento_utc, 'America/Sao_Paulo') AS ts_evento_br,
    to_date(data_referencia_ext, 'dd-MM-yyyy')            AS data_ajustada,
    to_timestamp(data_sistema_ext, 'dd-MM-yyyy HH:mm:ss')   AS timestamp_ajustado,
    date_add('2024-01-01', 30)                             AS trinta_dias_depois,
    datediff('2024-12-31', '2024-01-01')                   AS diff_dias
FROM logs;
  • Padronize tipos antes de joins e MERGE INTO
  • Prefira TRY_CAST em origem Bronze e trate os NULLs gerados conscientemente
  • Declare DECIMAL(p, s) para valores monetários
  • Documente a regra de timezone da pipeline: sessão local, UTC de armazenamento e conversão de exibição

Condicionais e Preenchimento de Nulos

SQL

O que é

NULL não significa string vazia nem zero — é pura ausência de estado. Deixá-los vazar quebra agrupadores e derruba joins em painéis de BI.

Júnior / Pleno

COALESCE e CASE WHEN

SQL
SELECT
    id_produto,
    COALESCE(nome_produto, 'SEM NOME ORIGINAL')  AS descricao_valida,
    COALESCE(id_categoria, -1)                    AS id_categoria, -- FK default -1
    CASE
        WHEN status_atual = 'A'                         THEN 'ALTO_RISCO'
        WHEN status_atual = 'M' AND is_ativo = True     THEN 'MEDIO_RISCO'
        ELSE 'BAIXO_RISCO'
    END AS flag_risco
FROM base_catalogo;

Impacto Oculto em Contagens

⚠ COUNT(1) conta linhas físicas. COUNT(coluna) ignora NULLs. Nunca faça INNER JOIN entre duas chaves nulas — NULL = NULL retorna FALSO em SQL nativo.
Sênior

NULLIF e Quality Flags

SQL
SELECT
    b.valor_bruto,
    -- NULLIF evita divisão por zero: se comissao = 0, vira NULL (não crash)
    b.valor_bruto / NULLIF(a.comissao_fixa, 0)      AS fator_multiplicador,
    IFF(a.comissao_fixa IS NULL, 1, 0)               AS is_linha_com_furo
FROM fatos a;

O Perigo da Reavaliação no Spark (Catalyst)

🔴 CTEs e Views não materializam dados. Se você referenciar a mesma CTE pesada 3 vezes, o Spark lerá o parquet do disco 3 vezes. Use CACHE VIEW quando precisar reusar uma lógica custosa.
SQL
-- Força o cluster a segurar o frame em RAM para múltiplas referências
CACHE VIEW cte_cacheada AS SELECT * FROM fatos_trilionaria;

SELECT * FROM a JOIN cte_cacheada...
UNION ALL
SELECT * FROM b JOIN cte_cacheada...

O Terror: Exploding Join

🔴 Se a dimensão tiver mais de uma linha por chave, a fato é multiplicada silenciosamente e os números ficam incorretos. Sempre deduplique o lado direito com QUALIFY ROW_NUMBER() = 1 antes do join.
SQL
WITH Dimensao_Sanitizada AS (
    SELECT *
    FROM Lixo_Dimensional_B
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY id
        ORDER BY updatetime DESC
    ) = 1
)
SELECT ...
FROM fato_a
LEFT JOIN Dimensao_Sanitizada b ON fato_a.id = b.id;

Range Frames e OOM

🔴 Nunca use AVG(preco) OVER (ORDER BY mes) sem PARTITION BY. Sem partição, todo o cluster cai para um único nó, causando OOM em bases bilionárias.
SQL
-- Média móvel de 7 dias (range restrito)
SELECT
    vendedor, venda,
    AVG(venda) OVER (
        PARTITION BY vendedor ORDER BY d_movimento
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS media_mov_7dias
FROM financeiro;

ON target.id_venda_pk = source.id_venda_pk

def pct_nulo(df, col_name):
    return df.select(
        (sum(when(col(col_name).isNull(), 1).otherwise(0)) / count("*") * 100).alias("pct")
    ).collect()[0]["pct"]

if pct_nulo(df, "telefone") > 25.0:
    print("WARNING: >25% de telefones nulos")

Arquitetura Medalhão no Databricks

Bronze → Silver → Gold

O que é

A arquitetura medalhão organiza os dados em camadas com papéis distintos, separando ingestão, confiabilidade estrutural e consumo analítico.

🥉 Bronze

Dado cru. Preservar rastreabilidade. Schema próximo da origem. Metadados de ingestão. Sem regras de negócio.

🥈 Silver

Cast e tipagem. Tratamento de nulos. Limpeza de strings. Deduplicação. Chaves estabilizadas. Grão confiável.

🥇 Gold

Marts, fatos, dimensões. Regras de negócio consolidadas. Agregações. Pronto para BI, analytics e reporting.

Heurística de Decisão Rápida

  • A tabela existe para preservar a origem → Bronze
  • A tabela existe para tornar o dado confiável e reutilizável → Silver
  • A tabela existe para análise, KPI ou BI → Gold
⚠ Erros comuns: fazer limpeza destrutiva na Bronze, pular a Silver, agregar cedo demais e perder flexibilidade analítica, ou não explicitar o grão final da tabela Gold.
