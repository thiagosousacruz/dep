Framework de Domínio Técnico

Como aprender, praticar e masterizar qualquer tópico — exemplo: window functions

5 estágios para qualquer tópico técnico — exemplo: window functions

1
Entenda o conceito sem tocar em código
Construir o modelo mental antes de qualquer sintaxe
30–60 min ›

Antes de abrir um notebook, você precisa saber responder: o que essa coisa faz que não conseguiria fazer de outro jeito? Para window functions: a resposta é "calcula sobre um grupo sem colapsar as linhas — ao contrário do GROUP BY".

  • a Leia a documentação oficial (Spark ou SQL) — só a seção conceitual, não os exemplos ainda
  • b Desenhe num papel: "tenho 10 linhas de vendas, o que RANK() faz nelas?"
  • c Escreva com suas palavras o que é PARTITION BY, ORDER BY e ROWS/RANGE — sem copiar da doc
  • d Crie um card no Anki: frente = "diferença entre ROW_NUMBER, RANK e DENSE_RANK", verso = sua explicação com um exemplo numérico

⚠ Você não entendeu se só consegue repetir o que a doc diz. Você entendeu quando consegue prever o resultado antes de rodar.

2
Escreva do zero sem consultar
A primeira linha de código tem que ser sua, do zero
1–2 horas ›

Abra um notebook em branco. Crie um DataFrame ou tabela na mão. Tente escrever a query/código sem olhar nada. Vai travar — tudo bem. O travamento é o aprendizado acontecendo.

  • a Crie dados simples no próprio notebook — vendas por loja, corridas do Garmin, o que for
  • b Escreva: ranking de vendas por categoria, running total, média móvel de 3 períodos — do zero
  • c Quando travar, consulte só a sintaxe (documentação), nunca o "como resolver esse problema"
  • d Rode, veja o resultado, confirme que bate com o que você previu no estágio 1
-- Crie isso do zero, sem copiar:
SELECT
  produto,
  categoria,
  valor,
  RANK() OVER (
    PARTITION BY categoria
    ORDER BY valor DESC
  ) AS ranking,
  SUM(valor) OVER (
    PARTITION BY categoria
    ORDER BY valor DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS acumulado
FROM vendas

⚠ Se você copiou o código de algum lugar, refaça. A mão digitando o código que o cérebro está pensando é o que cria memória muscular.

3
Quebre e entenda os limites
Provocar erros de propósito para aprender o comportamento real
1–2 horas ›

Especialista conhece o comportamento nos casos difíceis, não só no caminho feliz. Aqui você vai deliberadamente provocar situações problemáticas e observar o que acontece.

  • a Coloque NULLs nos dados e observe: RANK ignora? LAG retorna NULL? O que muda com NULLS LAST?
  • b Troque ROWS por RANGE — o que muda quando há empates no ORDER BY?
  • c Remova o PARTITION BY — o que acontece? Quando isso é intencional?
  • d Use LAG com offset=2 e um default diferente de NULL — entenda cada parâmetro
  • e Anote cada descoberta: "quando X, acontece Y porque Z". Isso vai para o Anki
-- Quebre de propósito e observe:

-- 1. O que RANK faz com empates?
SELECT produto, valor,
  RANK() OVER (ORDER BY valor DESC) rank_com_empate,
  DENSE_RANK() OVER (ORDER BY valor DESC) dense_rank
FROM vendas  -- insira linhas com valor igual!

-- 2. ROWS vs RANGE com empates
SUM(valor) OVER (ORDER BY valor ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
SUM(valor) OVER (ORDER BY valor RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
-- resultado diferente! Por quê?

✓ Você passou nesse estágio quando consegue prever o comportamento antes de rodar — incluindo nos casos de borda.

4
Aplique num problema real seu
Conectar o conceito a um dado que você conhece
2–4 horas ›

Aqui o conceito encontra a realidade. Use dados da Renner ou do Garmin — algo onde você conhece o negócio e consegue validar se o resultado faz sentido. Problema inventado com dado inventado não treina o julgamento.

  • a Renner: rank de cobertura de estoque por subclasse, variação MoM de receita por categoria, top-3 SKUs por canal
  • b Garmin: pace médio móvel por semana, rank de corridas por distância dentro do mês, evolução do VO2max
  • c Não use o conceito como fim — use-o para responder uma pergunta de negócio real
  • d Revise o resultado com o olhar de quem conhece o dado: "esse número faz sentido?"

⚠ Se você precisou consultar a sintaxe para escrever, anote o que travou e volte ao estágio 2 com esse ponto específico. Não pule.

5
Ensine ou explique para alguém
O teste final de domínio real
1 hora ›

Se você consegue explicar para alguém o conceito do zero — sem consultar nada, com exemplos, e respondendo perguntas — você dominou. Antes disso, você só conhece.

  • a Explique window functions para um colega que não conhece — use um exemplo de negócio real
  • b Se não tiver colega disponível: escreva uma explicação técnica como se fosse para o time
  • c Use Claude como "aluno": peça para fazer perguntas difíceis e responda sem consultar nada
  • d Grave um vídeo curto explicando (nem que seja para você mesmo) — ver sua própria explicação revela os buracos

✓ Quando você consegue explicar ROWS vs RANGE, o comportamento com NULLs, e quando usar LAG vs LEAD sem hesitar — o tópico está dominado.

Onde você está — seja honesto, a progressão depende disso

0
Nunca vi isso
Nunca usou ou ouviu falar. Começa do estágio 1.
Início
1
Já vi / já copiei
Sabe que existe e já usou copiando de algum lugar, mas não consegue escrever do zero nem explicar o PARTITION BY sem consultar.
Reconhece
2
Consigo escrever consultando
Consegue montar uma window function consultando a doc de sintaxe, entende o resultado, mas trava nos casos de borda (ROWS vs RANGE, NULLs, sem PARTITION).
Funcional
3
Escrevo do zero e sei o porquê
Escreve sem consultar, prevê o resultado nos casos de borda, e consegue explicar a diferença entre RANK e DENSE_RANK com um exemplo de empate na cabeça.
Competente
4
Escolho a ferramenta certa e ensino
Dado um problema de negócio, sabe identificar que window function resolve (e qual), sabe quando ela é pior que GROUP BY, e consegue explicar para qualquer pessoa do time sem hesitar.
Especialista

A maioria para no nível 2 e acha que está no 3. O teste real é simples: feche tudo e escreva. Se travou na sintaxe, está no nível 2. Se escreveu mas não consegue explicar o comportamento com NULLs, está no nível 2. O nível 3 é o mínimo aceitável para um especialista.

O que separa quem domina de quem acha que domina

✏️
Sempre escreva antes de consultar
Tente escrever do zero, trave, anote onde travou. Só então consulte. A consulta antes de tentar cancela o aprendizado — você reconhece o código, não o produz.
Fazer
📋
Nunca copie e cole código de exemplo
Se você achou um exemplo bom, feche, e escreva o seu do zero com o mesmo padrão. Copiar não cria memória. Redigitar o que você entendeu, sim.
Não fazer
🗄️
Use dados reais ou verossímeis
Dados inventados sem contexto não treinam julgamento. Use dados do Garmin, ou crie um dataset que simule Renner. Quando o dado é real, você percebe quando o resultado está errado.
Fazer
🤖
Não peça para a IA escrever o código por você
Use Claude para revisar depois que você escreveu, explicar um comportamento que você não entendeu, ou fazer perguntas difíceis para testar seu domínio. Nunca para gerar o código que você deveria estar aprendendo.
Não fazer
🃏
Anki para o que precisa ficar automático
Não para decorar sintaxe — para internalizar comportamentos. "O que RANGE faz diferente de ROWS quando há empates?" é um card útil. "Sintaxe do OVER()" não é.
Fazer
⏱️
Não gaste mais de 20% do tempo "aprendendo"
Assistir vídeo, ler artigo, ouvir podcast — isso não é praticar. O aprendizado técnico acontece quando você está com as mãos no código e o cérebro tentando resolver algo. 80% do tempo tem que ser prática ativa.
Não fazer
🔁
Volte ao tópico em intervalos (espaçamento)
Depois de dominar um tópico, revise-o ativamente 3 dias depois, 1 semana, 1 mês. Não releia — tente reproduzir do zero. O que você não consegue reproduzir semanas depois, você não dominou.
